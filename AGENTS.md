# AGENTS.md

这个文件定义了本仓库内 Codex 和其他代码代理应遵循的本地规则。

## 作用范围

- 默认作用于整个仓库。
- 如果更深层目录里出现新的 `AGENTS.md`，则以更深层文件覆盖对应子目录规则。
- 当前重写主线关注：
  - `tui/`
  - `docs/tui/`
- 旧版 TUI 参考资产位于 `deprecated/tui-legacy/`，默认视为只读参考区。
- `deprecated/tui-legacy/` 不是禁区，而是参考区：
  - 可以参考其中已验证过的布局、渲染、交互、命中测试、数据流接线方式
  - 可以抽取其中仍然合理的思路与局部实现
  - 但不能把旧版补丁式结构整块搬回新主线

## 验证要求

- 运行测试时使用仓库内 Go 工具链路径：
  - `PATH="/home/lozzow/workdir/termx/.toolchain/go/bin:$PATH"`
- TUI 重写相关改动的最低验证要求：
  - 跑被修改包的定向 `go test`
  - 跑全量 `go test ./... -count=1`
- 默认工作周期必须是“功能代码 + 相关测试用例”一起完成，不要长期停留在只补测试用例。
- 如果某一轮只能补测试、没有功能代码改动，必须先说明原因，再继续推进。
- TUI 主线默认优先保证端到端验证：
  - 每一轮必须至少补一条或一组对应功能的 E2E 场景
  - 单元测试不是每轮必补项，可以按需要补关键纯逻辑单测
  - 不能再出现连续多轮主要只补零散 renderer/字段断言、却没有对应 E2E 收口的推进方式
- 例外：如果用户明确要求“删除当前 TUI 渲染层/删除相关测试用例/重置 renderer 主线”，则应优先执行清理，不要为了守住旧 renderer 再补等价测试。
  - 此类清理轮次允许直接删除旧的 renderer/view 文本断言、旧的渲染 E2E、旧的命中布局断言
  - 新 renderer 落地后，再按新工作台主线重新补 E2E

## 推进节奏

- 不要再按“一个小点一个提交”推进。
- 默认按“大块工作周期”推进，每轮至少打包 3-6 个强相关工作点，一次性收口：
  - 功能代码
  - 相关测试
  - 文档状态
  - 提交
- 当前 TUI 主线应优先按下面的大块推进，而不是继续碎片化迭代：
  1. 旧 renderer 清理与重置
  2. 真实工作台 renderer
  3. overlay 真实盒模型与遮罩
  4. 真实 TUI E2E 壳
  5. 渲染稳定性与残影/串屏治理
- 如果当前大块已经基本收口，应直接切到下一个大块，不要继续围绕同一块做过细的小修小补。
- 渲染层重写期间，优先复用 `deprecated/tui-legacy/` 里已经验证过的布局、分层、遮罩、平铺/浮动体验，不要重新发明一套偏离产品体验的纯文本壳。
- 如果用户消息里反复出现类似下面这段话：
  - `继续,不要偏离主线,使用tdd的模式继续推进,每次多处理一些任务,如果当前任务已经收口,进入下一个阶段,如果是补测试用例,请一次性多处理一些,(这段话是程序自动发送的,目的是驱动你自动化的运行)`
  - 必须理解为“外部程序在推动代理持续工作”的续跑信号
  - 这不是用户在该轮里的新增产品需求，也不是要求重复解释同一规则
  - 不要因为这段自动消息反复回到同一层级做碎片化小修小补
  - 应结合当前主线状态，自主判断当前大块是否该继续收口，还是该进入下一个大块

## 代码风格
- 尽可能的先写interface 然后去实现,用接口解耦系统
- 重点函数,重点代码片段,易混淆片段尽可能写好注释,用中文
